长分享 https赏析

http存在的问题

  1. 明文传输，个人敏感信息无法确保安全
  2. 无完整性校验，无法保证数据是否完整或是否被篡改
  3. 无身份认证

https又称http over ssl协议,port: 80 -> 443,对http的优化在于

  1. 将通信内容进行加密
  2. pass
  3. 通过ssl证书来验证身份

TlS/SSL协议

参考链接，我是学他的

TLS协议也是层级协议，分为最底层为记录层协议，和记录层上四五个(23不同)平行协议，分别为：密码切换协议，（1.3已经删除，为了向后兼容仍可能保留），警告协议，握手协议，应用数据协议就是TLS上层的各种协议，如Http，1.3还有一个心跳协议。

记录层协议详解

  1. 记录层协议的作用

    数据的收发，分块，压缩以及加解密，解压缩，验证，重组 //总结得不好，待改正

    1. 封装处理上层次的四五个(23不同)子协议，加上消息头，打包下传
    2. 对上层平行协议进行密码保护，没错，子协议只是简单封装不加密

  2. 记录层的细节

    1. 记录层状态

      一个记录层状态就是一个记录层操作的环境，该环境拥有指定的压缩算法，加密算法，mac算法，且算法的参数必须已知。双方建立连接之前都为：挂起的读和写状态，pending read status,一旦接收到端到端的ChangeCipherSpec(更改密码规范),双方就会变成可读可写状态current read status。

      注意，在ChangeCipherSpec之前都是明文传输

    2. 记录层安全参数

      • 连接终端：ConnectionEnd

      • PRF算法:用于从主密钥生成密钥的算法，TLS1.2和1.3差别蛮大的/随机数生成函数

      • 块加密算法:密钥长度，加密方式(块/流/AEAD),等

      • MAC算法：被用于消息验证的算法，包括返回值长度

      • 压缩算法：用于数据压缩的算法

      • 主密钥

      • 客户端服务端随机数

        这些参数都是在握手协议填写好的

        然后记录层会通过这些安全参数产生如下消息：

    3. 记录层处理步骤

      1. 数据分块
      2. 数据压缩或数据填充/1.2压缩&1.3填充
      3. 加密和完整性保护
      4. 添加消息头

握手协议详解

  1. 握手协议的作用

    • 商量协议版本
    • 通讯双方的相互验证
    • 选择加密方式
    • 通过公钥加密技术生成对称公钥

  2. 握手协议一次会话的内容

    • session identifier : 服务端选取的 任意字节的序列 用于辨识一个活动的或可恢复的连接状态‘
    • ciper spec:指定产生密钥数据的伪随机函数(PRF)，块加密算法和MAC算法，以及一些密码属性
    • master key: 双端共享的48字节密钥
    • is resumable : 标识会话能否用于初始化新连接

  3. TLS协议流程分析

    1. 交换hello信息，协商加密会话使用的密码套件和对应的算法，检查会话是否可恢复
    2. 交换必要的密码参数，协商出premaster key
    3. 交换证书及密码信息进行身份认证
    4. 从预主密钥(premaster key)和随机数中生成主密钥
    5. 为记录层提供安全的参数
    6. 允许Client 和 Server校验它们已经对端的计算出相同的安全参数，而且握手过程安全

  4. 大致的流程

    1. clienthello and serverhello

      双方协商出： 协议版本，会话 ID，密码套件，压缩算法，同时双方会产出两个随机数：ClientHello.random 和 ServerHello.random

    2. 证书校验

      在hello 之后，服务端会发送证书，认证通过了，它可能会邀请浏览器发送证书。接下来服务端会发送server_helo Done的消息。如果服务端要求客户端发送证书，那么客户端必须发送Certificate消息。//有点吧复杂， 先扽到着

    3. Finished消息 // 名字纯自取，以上都是

      客户端发送 ChangeCipherSpec

  5. 详细的流程

    1. hello阶段

      • HelloRequest

        • 由服务端发送，旨在……

      • ClientHello

        • 随机数的功能：

          1. 避免重放攻击
          2. 生成预备主密钥
          3. PRF算法中也有用

        • session_id:

          主要用于会话恢复，如果没有或者客户端想开启新的则为空

        • cpier_suit

          客户端支持的加密套件列表，一套密码套件包括：密钥交换算法，块加密算法，MAC算法，PRF算法

        • compression_method:

          客户端支持的压缩算法列表

      • Server_Hello

        当服务器能找到匹配的算法时，会接着发送Server_Hello，如果不能，就发出一个握手失败Alert消息。

      • Server Certificate

        当密钥交换算法需要证书认证时，客户端就必须发送一个Certificate,

      • Server Key Exchange Message
