每次都会更新token用计算的时间换存储的空间

每次都会更新token

用计算的时间换存储的空间

让用户知道请求的是谁及状态

原因 — http 无状态

为什么无状态：

cookie是什么 http 里面的字段，键值对形式/单个值 免输入密码，跨站请求伪造攻击

session是什么 一种理念 ， 保存用户信息的模式，拿到session能直接登录

token 令牌； 将信息全部打包到token里面，前端返回给后端，后端解包，不用再服务器存储内容，也可以跨浏览器。

拿到token 能不能直接去登录

问题： 为什么有18个，过期了不会删除吗

cookie 数据载体

http请求 - server set cookie(id value) - sent to client -

session 服务器主导

session - (id-maxtime) 存储在数据库 - set cookie - add session(id-maxtime) - sent to client - save session (id-time)

用户名&密码 —- 对应session (id-maxtime)

redis —- 缓存

Jwt - json web token 客户端主导

client 首次登录 - 发送用户名&密码

服务器生成jwt - 服务器只保存jwt签名的密文，发送jwt给client

client 存储，cookie / storage

以后只需要发送jwt

jwt不用放内存，session需要放在内存

jwt无状态，不用存储再内存或者db,不用另外存储过期时间

session maxtime : 节约内存 & 过期保证安全

会员怎么确定

引例：当我们访问某个网页的时候，输入网址，如果没有登录，大部分网站都会自己弹出登录选项，而当我们登录之后，很长时间都不用再输入密码，那么是什么让服务器知道浏览器的状态

首先是cookie：

当浏览器首次请求客户端后，用户登录成功，服务器set-cookie，并跟随响应头发送给客户端，客户端保存在内存当中，下一次访问的时候，附加在请求头上，服务端根据客户端发送来的cookie判断是否需要重新登录

cookie格式：

cookie 存在的问题：

cookie就相当于一个钥匙，任何人只要有钥匙，都可去开门。但钥匙会丢。

实验：不同浏览器的cookie不同，把谷歌cookie直接在edge上面登录，也可以直接登录，大大的不安全啊

问题在于： cookie到底是直接存储用户名密码还是加密存储，如果加密，破解难度一定不高吗

自动填充账号 & 密码

根据cookie破译出密码 //后话

cookie是一种存储方式，session是编程方式

再就是session：

浏览器首次请求之后，服务器直接生成session-id & session-maxtime 等等信息，跟随cookie发送到浏览器，此时

浏览器保存session id 等内容，下一次登录把这些直接发送给服务器，服务器一看老熟人了。

优点： maxtime

缺点：任然可以拿去直接用

所以就有了token: 令牌虎符

浏览器首次请求之后，服务器直接生成token，并将此token发送给客户端，uid + time + sign 存储在cookie / 其他

相比于session : 可以跨服务器 也可以解决跨站请求伪造 ，解决cookie跨域

Jwt

  • 有一套自己的规则，
  • 无状态，不用再服务器存储内容
  • 不用单独存储过期时间

三者区别与联系： 三者都是因为http无状态，用来记录用户状态的。cookie是载体，session & token是一种解决问题的方式：

后两者的思想是

无状态 & 有状态

实验：

写一个小网站，set_cookie & charge_cookie

反思：层层增加，安全机制
